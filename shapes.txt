````md
# Forge Shape — Plan

## Overview

Forge Shape is the project’s **current structural map**.

It exists to help the agent build and maintain a clean mental model of:

- what the user or operator encounters
- what the system can do
- what major internal pieces support those outcomes
- how the project currently fits together

Forge Shape is a structural layer, not a reasoning assistant and not a note system.

It should help the agent say:

> “I understand how this project is currently shaped.”

Not:

> “I have a lot of notes.”

And not:

> “I know the code internals in detail.”

---

## Why Forge Shape exists

Without Shape, important structural understanding tends to get scattered across:

- repo files
- handoff summaries
- phase notes
- memory entries
- implicit agent understanding

That makes it harder for later sessions to answer questions like:

- what are the main user-facing areas?
- what are the main capabilities of the project?
- what major pieces support those capabilities?
- how is the project expected to behave from the outside?
- what has become structurally true about the project now?

Forge Shape gives that understanding one dedicated home.

---

## The problem it solves

Forge already covers other layers:

### `plan`
What the project is intended to be.

### `memory`
What happened, changed, or remains unresolved.

### `phases`
How work progresses through time.

But there is still a missing middle layer:

### `shape`
What the project currently looks like as a working system.

This is the project’s lived structure:
- the user-facing form
- the functional form
- the high-level internal support form

That is what Shape solves.

---

## Separation of concerns

## `plan`
Use for:
- stack
- product direction
- design posture
- architectural principles
- stable intended identity

Question it answers:
> What is this project supposed to be?

## `shape`
Use for:
- surfaces
- capabilities
- parts
- current structural form

Question it answers:
> How does this project currently fit together?

## `memory`
Use for:
- decisions
- issues
- concerns
- interpretations
- superseded conclusions
- handoff state
- observations

Question it answers:
> What happened to this project over time, and what changed in understanding?

## `phases`
Use for:
- execution progress
- phase sequence
- step completion

Question it answers:
> Where is the work, and how has it progressed?

---

## Core design principle

Forge Shape should be:

- structural, not verbose
- durable, not fleeting
- useful for orientation
- useful for resumption
- useful for understanding how the project works together
- sparse enough that it stays readable

If a Shape entry does not help a later session understand the project faster and more correctly, it probably does not belong there.

---

## Shape v1 model

Shape v1 uses exactly three domains:

- `surfaces`
- `capabilities`
- `parts`

This is the leanest cut that still forms a meaningful project map.

`.forge/shape.json`
````

## Domain 1: Surfaces

### What Surfaces are

Surfaces describe the project as it is experienced from the outside.

This includes:

* screens
* views
* panels
* entry points
* visible outputs
* primary user-facing regions
* expected flow through those places

Flow belongs inside surfaces.

A surface should describe:

* what the user sees there
* what they are trying to do there
* what stands out first
* what actions feel primary
* what appears, disappears, or becomes contextual
* where the user can go from there

### What belongs in Surfaces

Good surface details:

* “Dashboard is the main triage surface.”
* “Users typically start with filters before opening a case.”
* “The summary panel becomes meaningful after case selection.”
* “Back appears only inside detail contexts.”
* “Empty state pushes the user toward first-item creation.”

### What does not belong in Surfaces

Avoid:

* exact DOM detail
* CSS implementation detail
* code internals
* architecture commentary
* generic product notes that do not describe actual experience

## Domain 2: Capabilities

### What Capabilities are

Capabilities describe what the system can meaningfully do.

They are not screens and not internal pieces.
They are the system’s functional powers.

Examples:

* filtering
* summary generation
* case movement
* note editing
* undo
* export
* authentication
* import

A capability should describe:

* what problem it solves
* what it enables the user to accomplish
* what inputs it works from
* what outputs or state changes it creates
* where it is surfaced
* what parts support it

### What belongs in Capabilities

Good capability details:

* “Filtering narrows visible work by program, due window, and query.”
* “Summary generation transforms selected case state into bounded outputs.”
* “Undo restores the most recent status move.”

### What does not belong in Capabilities

Avoid:

* exact UI placement
* exact implementation details
* code symbol names
* vague aspirational feature language

Parts are the major built pieces that make the project possible.

This includes:

* internal modules
* high-level services
* major subsystems
* important storage/runtime layers
* important libraries or platforms when structurally relevant

Parts should describe:

# Tool schema: `forge_update_shape`

This is the dedicated Shape mutation tool.

## Purpose

Make explicit structural updates to:

* `surfaces`
* `capabilities`
* `parts`

This is for shape-worthy changes, not tiny edits.

---

## Design choice

I recommend **multi-operation input** so one tool call can update several shape entries at once.

That keeps the tool powerful without multiplying tool calls.

---


# Recommended default behavior rules

## `forge_shape`

Default behavior:

* all domains if none specified
* `compact` mode if no mode specified
* include summary and counts by default

## `forge_update_shape`

Default behavior:

* update `updated_at` on every successful write
* preserve existing `shape_version`
* patch should fail clearly if the target `id` does not exist
* upsert should insert or replace by `id`
* remove should fail clearly if the target `id` does not exist
