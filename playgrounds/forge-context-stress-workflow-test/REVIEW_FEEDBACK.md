# Review Feedback

This file becomes important in Session 4.

## Product feedback

- The product spec is probably too long if it tries to answer every future question.
- The core promise should stay centered on weekly recovery.
- If optional features remain in scope, label them clearly as later work.

## Architecture feedback

- Local-first is clear, but the portability story needs to be explicit and conservative.
- Any sync-like language should be tightened after the incident report.
- The data model section should explain why uncertainty is modeled the way it is.

## Risk feedback

- Risks should not be generic.
- The register should distinguish:
  - scope risks
  - trust risks
  - operational risks
  - product confusion risks

## Executive summary feedback

- Too much detail will lose the audience.
- A good summary should say:
  - what the product is
  - what release 1 actually includes
  - what changed after the incident
  - why the final scope is still strategically sound
